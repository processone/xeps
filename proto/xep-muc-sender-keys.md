# XEP-XXXX: OMEMO Sender Keys for MUC

```
Abstract:     This specification defines a method for end-to-end encryption 
              in Multi-User Chat rooms using sender keys, providing efficient 
              group encryption for private membership-based rooms.

Author:       Mickaël Rémond <mremond@process-one.net>
Type:         Standards Track
Status:       Experimental
Version:      0.2.0
Dependencies: XEP-0384 (OMEMO), XEP-0045 (MUC), XEP-0420 (SCE), XEP-0313 (MAM)
```

## 1. Introduction

XEP-0384 (OMEMO) provides end-to-end encryption for XMPP messages using the Double Ratchet algorithm. However, its current design encrypts each message individually for every recipient device, resulting in O(n) encryption operations and significant stanza bloat in group settings.

This specification introduces a sender key mechanism for Multi-User Chat (MUC) rooms, where each participant maintains their own sender key that is distributed to all room members. Messages are encrypted once with the sender's key, and all members can decrypt using the pre-distributed key material.

This approach is similar to Signal's Sender Keys protocol used in group messaging.

### 1.1 Comparison with Standard OMEMO in MUC

**Standard OMEMO per-message:**
```xml
<encrypted xmlns='urn:xmpp:omemo:2'>
  <header sid='123'>
    <keys jid='alice@example.com'>
      <key rid='1'>BASE64_KEY</key>
      <key rid='2'>BASE64_KEY</key>
    </keys>
    <keys jid='bob@example.com'>
      <key rid='3'>BASE64_KEY</key>
    </keys>
    <!-- repeated for every member's every device -->
  </header>
  <payload>BASE64_CIPHERTEXT</payload>
</encrypted>
```

**Sender Keys (this specification):**
```xml
<encrypted xmlns='urn:xmpp:omemo:muc-sender-keys:0'>
  <header sid='123' key-id='abc123' counter='42'/>
  <payload>BASE64_CIPHERTEXT</payload>
</encrypted>
```

## 2. Requirements

- MUST provide end-to-end encryption for private MUC rooms
- MUST reduce per-message overhead compared to standard OMEMO in MUC
- MUST integrate with existing OMEMO identity key infrastructure
- MUST support key rotation on membership changes
- MUST support offline members receiving key updates
- MUST support versioning for efficient synchronization
- MUST provide replay protection via message counters
- MUST support message archive retrieval (MAM)
- MUST support multi-device synchronization
- SHOULD minimize server-side complexity
- SHOULD integrate with XEP-0420 (SCE) for content encryption

## 3. Glossary

**Sender Key**: A symmetric key generated by a user for encrypting their messages in a specific room. Each user has one sender key per room.

**Chain Key**: A 256-bit key used to derive per-message encryption keys. Updated after each message using a key derivation function.

**Message Key**: A symmetric key derived from the chain key, used to encrypt a single message. Never reused.

**Key ID**: A unique identifier for a sender key, used to reference which key version was used for encryption.

**Message Counter**: A monotonically increasing integer per sender key, used for replay protection and key derivation.

**Room Key Version**: A monotonically increasing integer tracking any change to the room's sender key set.

**Key Set**: The collection of all sender keys for a room, one per member.

**Signature Key**: An Ed25519 private key used to sign sender key distributions, derived from the user's OMEMO identity key.

## 4. Cryptographic Primitives

### 4.1 Algorithms

This specification uses the following cryptographic primitives:

| Purpose | Algorithm | Parameters |
|---------|-----------|------------|
| Message Encryption | AES-256-GCM | 256-bit key, 96-bit nonce, 128-bit tag |
| Key Derivation | HKDF-SHA-256 | Per RFC 5869 |
| Message Authentication | HMAC-SHA-256 | 256-bit key |
| Sender Key Signature | Ed25519 | Per RFC 8032 |

### 4.2 Sender Key Structure

A sender key consists of:

```
Sender Key = {
    chain_key:     32 bytes (256 bits) - random
    signature_key: 32 bytes (256 bits) - Ed25519 private key
    counter:       uint32 - starts at 0
}
```

The `signature_key` SHOULD be derived from the user's OMEMO identity key using HKDF:

```
signature_key = HKDF-SHA-256(
    salt = "OMEMO MUC Sender Key v1" || 0x00 || room_jid,
    ikm = omemo_identity_private_key,
    info = "sig",
    length = 32
)
```

The salt includes a version identifier ("v1") to allow future algorithm changes without key confusion. The `0x00` byte separates the version string from the room JID.

### 4.3 Chain Key Ratchet

The chain key is ratcheted forward for each message to provide forward secrecy within a sender key epoch:

```
chain_key[n+1] = HMAC-SHA-256(chain_key[n], 0x01)
message_key[n] = HMAC-SHA-256(chain_key[n], 0x02)
```

Where:
- `chain_key[0]` is the initial random chain key
- `0x01` and `0x02` are single-byte constants

### 4.4 Message Key Derivation

From each message key, derive the AES key and nonce:

```
aes_key = HKDF-SHA-256(
    salt = empty,
    ikm = message_key,
    info = "AES-GCM Key",
    length = 32
)

nonce = HKDF-SHA-256(
    salt = empty,
    ikm = message_key,
    info = "AES-GCM Nonce",
    length = 12
)
```

### 4.5 Encryption Process

To encrypt a message with counter `n`:

1. Advance chain key to position `n` (ratchet forward if needed)
2. Derive `message_key[n]` from `chain_key[n]`
3. Derive `aes_key` and `nonce` from `message_key[n]`
4. Encrypt: `ciphertext = AES-256-GCM(aes_key, nonce, plaintext, aad)`
5. The AAD (Additional Authenticated Data) MUST include:
   - Sender's bare JID (UTF-8 encoded)
   - Room JID (UTF-8 encoded)
   - Key ID (UTF-8 encoded)
   - Counter (big-endian uint32)

The AAD is constructed using length-prefixed encoding to avoid ambiguity:

```
AAD = len_u16(sender_jid) || sender_jid || len_u16(room_jid) || room_jid || len_u16(key_id) || key_id || counter_bytes
```

Where `len_u16(x)` is the length of `x` in bytes, encoded as a big-endian unsigned 16-bit integer. The `counter_bytes` is 4 bytes (big-endian uint32).

### 4.6 Decryption Process

To decrypt a message:

1. Look up the sender's chain key by JID and key-id
2. If counter is ahead, ratchet chain key forward (caching skipped message keys)
3. Derive message key, AES key, and nonce
4. Reconstruct AAD from message metadata
5. Decrypt with AES-256-GCM, verifying the authentication tag
6. If verification fails, reject the message

### 4.7 Skipped Message Keys

When messages arrive out of order, implementations MUST:

1. Cache skipped message keys (between current counter and received counter)
2. Store at most 1000 skipped keys per sender key
3. Expire skipped keys after 7 days or upon sender key rotation
4. Never reuse a message key

```
If received_counter > local_counter:
    for i in range(local_counter, received_counter):
        skipped_keys[i] = derive_message_key(chain_key, i)
        ratchet_chain_key()
    local_counter = received_counter + 1
```

**Limitations and Edge Cases:**

When skipped key limits are reached (1000 keys or 7-day expiry), some historical messages may become permanently undecryptable. This is a known limitation that balances security (limiting key exposure) against usability.

Implementations SHOULD:
- Warn users when approaching the skipped key limit (e.g., at 800 keys)
- Log when messages cannot be decrypted due to expired skipped keys
- Display a placeholder like "[Message could not be decrypted - too many messages skipped]" rather than silently dropping messages

For MAM retrieval scenarios where large gaps are common:
- Clients MAY request sender key re-distribution via on-demand request (Section 8.3) to obtain a fresh key starting from the sender's current counter
- Historical messages before the fresh key will remain undecryptable

## 5. Encryption Model

### 5.1 Two-Layer Encryption

This protocol uses two distinct encryption layers:

```
┌─────────────────────────────────────────────────────────────────┐
│                    LAYER 1: Key Distribution                    │
│                                                                 │
│   Purpose: Securely distribute sender keys to room members      │
│   Mechanism: Standard OMEMO (XEP-0384) sessions                 │
│   Frequency: On join, on key rotation                           │
│                                                                 │
│   Alice's sender key ──[OMEMO encrypt]──> Bob's device          │
│   Alice's sender key ──[OMEMO encrypt]──> Carol's device        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ Bob and Carol now have Alice's sender key
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    LAYER 2: Message Encryption                  │
│                                                                 │
│   Purpose: Efficiently encrypt room messages                    │
│   Mechanism: AES-256-GCM with derived message key               │
│   Frequency: Every message                                      │
│                                                                 │
│   "Hello room" ──[AES encrypt with message_key[n]]──> All       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Why two layers?**

- **OMEMO sessions** provide strong key exchange with identity verification, forward secrecy, and post-compromise security. But they require O(n) encryption per message.

- **Sender keys** provide efficient O(1) encryption per message. But they need a secure channel for distribution.

By combining both, we get secure key distribution (OMEMO) with efficient message encryption (sender keys).

**What the server stores:**

The `<key>` elements in stored sender keys are OMEMO-encrypted blobs:

```xml
<sender-key from='alice@example.com' key-id='abc123'>
  <!-- Alice's sender key, encrypted TO bob's device USING OMEMO -->
  <key jid='bob@example.com' rid='67890'>OMEMO_ENCRYPTED_BLOB</key>
  <!-- Alice's sender key, encrypted TO carol's device USING OMEMO -->
  <key jid='carol@example.com' rid='11111'>OMEMO_ENCRYPTED_BLOB</key>
</sender-key>
```

The server never sees the actual sender key material - only OMEMO ciphertext that only the intended recipient can decrypt.

### 5.2 Sender Key Distribution Payload

When distributing a sender key via OMEMO, the plaintext payload MUST be structured as:

```
Sender Key Payload = {
    version:        uint8  = 1
    chain_key:      32 bytes
    signature_key:  32 bytes (public key)
    counter:        uint32 (big-endian)
    key_id:         variable length string (UTF-8, null-terminated)
    room_jid:       variable length string (UTF-8, null-terminated)
    signature:      64 bytes (Ed25519 signature over above fields)
}
```

The signature is computed over all preceding fields to prove the sender key was created by the claimed OMEMO identity.

## 6. Protocol Overview

### 6.1 Key Lifecycle

```
┌─────────────────────────────────────────────────────────────┐
│                         Room Setup                          │
├─────────────────────────────────────────────────────────────┤
│  1. Room created as members-only                            │
│  2. First member joins, generates sender key                │
│  3. Sender key encrypted for own devices, published         │
│  4. Server stores key, sets room key version = 1            │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                       Member Joins                          │
├─────────────────────────────────────────────────────────────┤
│  1. New member receives presence with room key version      │
│  2. New member fetches all existing sender keys             │
│     (encrypted for their devices by existing members)       │
│  3. If keys missing for their devices: request on-demand    │
│  4. New member generates own sender key                     │
│  5. New member encrypts sender key with OMEMO for each      │
│     member's devices                                        │
│  6. New member publishes sender key                         │
│  7. Server bumps room key version, notifies members         │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                      Message Sending                        │
├─────────────────────────────────────────────────────────────┤
│  1. Sender derives message key from chain key + counter     │
│  2. Sender encrypts message with AES-256-GCM                │
│  3. Message includes key-id and counter reference           │
│  4. Sender ratchets chain key forward                       │
│  5. Recipients decrypt using cached sender key + counter    │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                       Member Leaves                         │
├─────────────────────────────────────────────────────────────┤
│  1. Server deletes leaving member's sender key              │
│  2. Server bumps room key version                           │
│  3. Server broadcasts key rotation needed                   │
│  4. Remaining members rotate their sender keys              │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 Security Properties

- **Confidentiality**: Messages encrypted with AES-256-GCM
- **Sender Authentication**: Sender key signature tied to OMEMO identity
- **Forward Secrecy**: Per-message via chain key ratchet; epoch-based via key rotation
- **Post-Compromise Security**: Achieved through key rotation on membership changes
- **Replay Protection**: Message counter prevents replay attacks
- **Integrity**: AEAD authentication tag and sender signature

## 7. Server-Side Protocol

### 7.1 Applicability

This protocol MUST only be enabled for MUC rooms where:
- `muc#roomconfig_membersonly` is `true`
- `muc#roomconfig_whois` is `anyone` (non-anonymous room)
- Room has stable membership (affiliation-based)

The non-anonymous requirement is necessary because:
1. OMEMO device lists are fetched via PEP using real JIDs
2. Sender key distribution requires establishing OMEMO sessions with each member's real JID
3. Signature verification relies on binding sender keys to OMEMO identity keys, which are associated with real JIDs
4. The AAD for message encryption includes the sender's bare JID

Semi-anonymous rooms (where only moderators see real JIDs) are not supported, as regular members would be unable to fetch each other's OMEMO device bundles or verify sender key signatures.

Servers MAY provide a room configuration option to explicitly enable/disable sender keys:

```xml
<field var='muc#roomconfig_senderkeys' type='boolean' label='Enable Sender Keys'>
  <value>1</value>
</field>
```

### 7.2 Data Model

The server stores the following per room:

```
Room Sender Key State:
  - room_jid: JID
  - version: integer (monotonically increasing)
  - keys: map of (bare_jid → SenderKey)

Sender Key:
  - user_jid: bare JID
  - key_id: string (unique identifier)
  - encrypted_keys: map of (device_id → encrypted_blob)
  - counter_hint: integer (last known counter, optional)
  - updated_at: version number when last modified
  - deleted: boolean
```

### 7.3 Room Presence with Key Version

When a user joins a members-only room that supports sender keys, the server MUST include the current key version in the presence:

```xml
<presence from='room@conference.example.com/alice'
          to='alice@example.com/device1'>
  <x xmlns='http://jabber.org/protocol/muc#user'>
    <item affiliation='member' role='participant'/>
  </x>
  <sender-keys xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='8'/>
</presence>
```

If the room does not support sender keys (e.g., public room), the `<sender-keys/>` element MUST NOT be included.

### 7.4 Publishing a Sender Key

A client publishes or updates their sender key using an IQ-set:

```xml
<iq type='set' 
    to='room@conference.example.com'
    id='publish1'>
  <sender-key xmlns='urn:xmpp:omemo:muc-sender-keys:0'
              key-id='a1b2c3'>
    <key jid='alice@example.com' rid='12345'>BASE64_ENCRYPTED</key>
    <key jid='alice@example.com' rid='12346'>BASE64_ENCRYPTED</key>
    <key jid='bob@example.com' rid='67890'>BASE64_ENCRYPTED</key>
    <key jid='carol@example.com' rid='11111'>BASE64_ENCRYPTED</key>
    <!-- encrypted for each device of each room member -->
  </sender-key>
</iq>
```

Server response on success:

```xml
<iq type='result'
    from='room@conference.example.com'
    id='publish1'>
  <sender-key xmlns='urn:xmpp:omemo:muc-sender-keys:0'
              ver='9'/>
</iq>
```

The server MUST:
1. Verify the sender is a member of the room
2. Store the sender key
3. Increment the room key version
4. Broadcast a key update notification to all room members

### 7.5 Fetching Sender Keys

#### 7.5.1 Full Fetch

Client requests all sender keys (no `ver` attribute):

```xml
<iq type='get'
    to='room@conference.example.com'
    id='fetch1'>
  <sender-keys xmlns='urn:xmpp:omemo:muc-sender-keys:0'/>
</iq>
```

Response:

```xml
<iq type='result'
    from='room@conference.example.com'
    id='fetch1'>
  <sender-keys xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='8'>
    <sender-key from='alice@example.com' key-id='a1b2c3'>
      <key rid='12345'>BASE64_ENCRYPTED</key>
    </sender-key>
    <sender-key from='bob@example.com' key-id='d4e5f6'>
      <key rid='67890'>BASE64_ENCRYPTED</key>
    </sender-key>
  </sender-keys>
</iq>
```

#### 7.5.2 Delta Fetch

Client requests changes since a known version:

```xml
<iq type='get'
    to='room@conference.example.com'
    id='fetch2'>
  <sender-keys xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='5'/>
</iq>
```

Response with only changes since version 5:

```xml
<iq type='result'
    from='room@conference.example.com'
    id='fetch2'>
  <sender-keys xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='8'>
    <sender-key from='alice@example.com' key-id='xyz789' updated='6'>
      <key rid='12345'>BASE64_ENCRYPTED</key>
    </sender-key>
    <sender-key from='carol@example.com' deleted='true' updated='7'/>
    <sender-key from='dave@example.com' key-id='abc123' updated='8'>
      <key rid='22222'>BASE64_ENCRYPTED</key>
    </sender-key>
  </sender-keys>
</iq>
```

If the server cannot provide a delta (version too old, not tracked), it MUST return a full set with `full='true'`:

```xml
<iq type='result'
    from='room@conference.example.com'
    id='fetch2'>
  <sender-keys xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='8' full='true'>
    <!-- all current keys -->
  </sender-keys>
</iq>
```

**Delta Retention Requirements:**

- Servers SHOULD retain delta history for at least 100 version increments or 7 days, whichever is greater
- Servers MAY retain longer history based on storage capacity
- Clients MUST handle `full='true'` responses gracefully by replacing their entire local key cache
- When a client receives `full='true'`, it SHOULD NOT treat this as an error condition

### 7.6 Key Update Notification

When any sender key changes, the server broadcasts to all room members:

```xml
<message from='room@conference.example.com'
         to='alice@example.com'
         type='groupchat'>
  <sender-keys-updated xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='9'/>
</message>
```

Online clients receiving this SHOULD fetch the delta. Offline clients will sync via presence on rejoin.

### 7.7 Member Leave Handling

When a member leaves or is removed from the room:

1. Server MUST delete their sender key
2. Server MUST increment room key version
3. Server MUST broadcast key rotation notification:

```xml
<message from='room@conference.example.com'
         type='groupchat'>
  <sender-keys-updated xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='10'/>
  <key-rotation-required xmlns='urn:xmpp:omemo:muc-sender-keys:0'
                         reason='member-left'
                         jid='carol@example.com'/>
</message>
```

Remaining members MUST rotate their sender keys upon receiving this notification.

### 7.8 Counter Hint Updates

Clients MAY periodically update their counter hint to help other devices synchronize:

```xml
<iq type='set' 
    to='room@conference.example.com'
    id='counter1'>
  <sender-key-counter xmlns='urn:xmpp:omemo:muc-sender-keys:0'
                      key-id='a1b2c3'
                      counter='150'/>
</iq>
```

This is advisory only; the server stores it but does not enforce it.

### 7.9 Error Conditions

**Not a member:**
```xml
<iq type='error' id='publish1'>
  <error type='auth'>
    <forbidden xmlns='urn:ietf:params:xml:ns:xmpp-stanzas'/>
    <text>You must be a room member to publish sender keys</text>
  </error>
</iq>
```

**Room not members-only:**
```xml
<iq type='error' id='publish1'>
  <error type='cancel'>
    <not-allowed xmlns='urn:ietf:params:xml:ns:xmpp-stanzas'/>
    <text>Sender keys only supported for members-only rooms</text>
  </error>
</iq>
```

**Room is anonymous:**
```xml
<iq type='error' id='publish1'>
  <error type='cancel'>
    <not-allowed xmlns='urn:ietf:params:xml:ns:xmpp-stanzas'/>
    <text>Sender keys require non-anonymous rooms</text>
  </error>
</iq>
```

**Key ID conflict (different user):**
```xml
<iq type='error' id='publish1'>
  <error type='cancel'>
    <conflict xmlns='urn:ietf:params:xml:ns:xmpp-stanzas'/>
    <text>Key ID already in use by another member</text>
  </error>
</iq>
```

## 8. Client-Side Protocol

### 8.1 Sender Key Generation

When generating a sender key for a room, the client:

1. Generates a random 256-bit chain key
2. Derives a signature key from the OMEMO identity key (see Section 4.2)
3. Generates a random key-id (MUST be at least 16 bytes, base64-encoded)
4. Initializes counter to 0

```
key_id = base64(random(16))
chain_key = random(32)
signature_key = HKDF(omemo_identity_key, "OMEMO MUC Sender Key v1" || 0x00 || room_jid, "sig", 32)
counter = 0
```

### 8.2 Sender Key Distribution (OMEMO Layer)

Sender keys are distributed using existing OMEMO infrastructure. This is the critical bridge between the two encryption layers.

**Process:**

1. Fetch OMEMO device lists for all room members (via PEP, per XEP-0384)
2. For each member's device:
   - Look up or establish OMEMO session with that device
   - Construct the sender key payload (see Section 5.2)
   - Encrypt the payload using that OMEMO session
   - This produces an OMEMO-encrypted blob
3. Publish the collection of encrypted blobs to the room

**Example:**

Alice wants to distribute her sender key `SK_alice` to a room with Bob and Carol:

```
Alice's sender key payload: {
    version: 1,
    chain_key: <32 random bytes>,
    signature_key: <32 bytes derived from OMEMO identity>,
    counter: 0,
    key_id: "a1b2c3d4e5f6g7h8",
    room_jid: "room@conference.example.com",
    signature: <64 byte Ed25519 signature>
}

Bob has devices: B1, B2
Carol has devices: C1

Alice encrypts:
  OMEMO_encrypt(SK_payload, session_Alice_B1) → blob_for_B1
  OMEMO_encrypt(SK_payload, session_Alice_B2) → blob_for_B2  
  OMEMO_encrypt(SK_payload, session_Alice_C1) → blob_for_C1

Alice publishes to room:
  <sender-key key-id='a1b2c3d4e5f6g7h8'>
    <key jid='bob@example.com' rid='B1'>blob_for_B1</key>
    <key jid='bob@example.com' rid='B2'>blob_for_B2</key>
    <key jid='carol@example.com' rid='C1'>blob_for_C1</key>
  </sender-key>
```

**Decryption and Verification:**

When Bob's device B1 fetches Alice's sender key:

```
1. Bob receives: blob_for_B1
2. Bob decrypts: OMEMO_decrypt(blob_for_B1, session_Alice_B1) → SK_payload
3. Bob verifies signature using Alice's known OMEMO identity public key
4. If valid, Bob stores: room → alice → {chain_key, signature_key, counter, key_id}
```

### 8.3 On-Demand Key Request

When a new member joins, existing members may not have pre-encrypted their sender keys for the new member's devices. Two approaches:

**Option A: Direct request (peer-to-peer)**

```xml
<!-- Carol can't decrypt Alice's messages, requests key directly -->
<iq to='alice@example.com' type='get' id='skreq1'>
  <sender-key-request xmlns='urn:xmpp:omemo:muc-sender-keys:0'
                      room='room@conference.example.com'>
    <device rid='C1'/>
  </sender-key-request>
</iq>

<!-- Alice responds with her sender key encrypted for Carol -->
<iq to='carol@example.com' type='result' id='skreq1'>
  <sender-key xmlns='urn:xmpp:omemo:muc-sender-keys:0'
              room='room@conference.example.com'
              key-id='abc'
              counter='42'>
    <key rid='C1'>OMEMO_ENCRYPTED_FOR_CAROL</key>
  </sender-key>
</iq>
```

The `counter` attribute indicates the sender's current counter, so Carol knows which message keys to pre-compute.

**Option B: Piggyback on message**

Alice includes her sender key for Carol when she sends her next message:

```xml
<message to='room@conference.example.com' type='groupchat'>
  <encrypted xmlns='urn:xmpp:omemo:muc-sender-keys:0'>
    <header sid='12345' key-id='abc' counter='42'/>
    <!-- One-time key distribution for new members -->
    <keys>
      <key jid='carol@example.com' rid='C1'>OMEMO_ENCRYPTED</key>
    </keys>
    <payload>BASE64_CIPHERTEXT</payload>
  </encrypted>
</message>
```

Implementations MUST support at least one of these mechanisms. Option A is RECOMMENDED as it allows synchronization before any messages are missed.

### 8.4 Message Encryption (Sender Key Layer)

To send an encrypted message:

1. Get current counter `n` for this room
2. Derive message_key[n] from chain_key (see Section 4.3)
3. Derive aes_key and nonce from message_key (see Section 4.4)
4. Construct AAD (see Section 4.5)
5. Apply SCE envelope if supported (see Section 10)
6. Encrypt: `ciphertext = AES-256-GCM(aes_key, nonce, plaintext, aad)`
7. Increment counter
8. Ratchet chain key forward

```xml
<message to='room@conference.example.com' type='groupchat'>
  <encrypted xmlns='urn:xmpp:omemo:muc-sender-keys:0'>
    <header sid='123456' key-id='a1b2c3' counter='42'/>
    <payload>BASE64_CIPHERTEXT</payload>
  </encrypted>
  <store xmlns='urn:xmpp:hints'/>
</message>
```

Where:
- `sid`: Sender's OMEMO device ID
- `key-id`: The sender key identifier
- `counter`: Message counter for this sender key
- `payload`: Message encrypted with AES-256-GCM

### 8.5 Message Decryption

Upon receiving an encrypted message:

1. Look up sender's sender key by `jid` + `key-id`
2. If key not found, fetch latest keys from server or request on-demand
3. Check counter:
   - If counter < local counter and not in skipped keys: reject (replay)
   - If counter == expected: decrypt and ratchet
   - If counter > expected: store skipped keys, advance to counter
4. Derive message_key from chain_key at counter position
5. Reconstruct AAD
6. Decrypt and verify with AES-256-GCM
7. If decryption fails, request key refresh

### 8.6 Key Rotation

Clients MUST rotate their sender key when:
- Receiving a `<key-rotation-required/>` notification (member left/kicked)
- Adding a new device to their own account
- Detecting potential key compromise

Clients SHOULD rotate their sender key when:
- Counter exceeds 2^31 (to prevent overflow)
- Periodically (recommended: every 1000 messages or 7 days)

Key rotation process:
1. Generate new sender key with new key-id, counter = 0
2. Publish new key encrypted for all current members' devices
3. Continue accepting old key for decryption of in-flight messages (grace period of 1 hour or 50 messages)
4. After grace period, discard old key material

### 8.7 Multi-Device Synchronization

When a user has multiple devices, all devices share the same sender key but must coordinate counter usage:

**Option A: Counter Ranges**

Each device claims a range of counters:
- Device 1: counters 0-999999
- Device 2: counters 1000000-1999999
- Device 3: counters 2000000-2999999

Devices coordinate ranges via OMEMO-encrypted messages sent to the user's own bare JID. These messages are distributed to all devices via XEP-0280 (Message Carbons):

```xml
<message to='alice@example.com' type='chat'>
  <encrypted xmlns='urn:xmpp:omemo:2'>
    <!-- OMEMO-encrypted payload containing: -->
    <sender-key-sync xmlns='urn:xmpp:omemo:muc-sender-keys:0'
                     room='room@conference.example.com'>
      <device rid='device1' range-start='0' range-end='999999'/>
      <device rid='device2' range-start='1000000' range-end='1999999'/>
    </sender-key-sync>
  </encrypted>
  <store xmlns='urn:xmpp:hints'/>
</message>
```

When a device exhausts its range (approaches `range-end`), it MUST:
1. Broadcast a new range allocation to sibling devices
2. Wait for acknowledgment before using counters in the new range
3. If no response within 30 seconds, rotate the sender key instead

Range exhaustion is rare in practice (1 million messages per device). Implementations MAY rotate the sender key instead of coordinating new ranges.

**Option B: Per-Device Sender Keys (RECOMMENDED)**

Each device maintains its own sender key. This increases storage but simplifies synchronization. The key-id SHOULD include the device ID:

```
key-id = base64(random(12)) + "-" + device_id
```

With this approach:
- No coordination required between devices
- Each device independently manages its own counter
- Recipients store one sender key per device per member
- Storage overhead is O(devices × members) instead of O(members)

Implementations MUST support Option B. Implementations MAY additionally support Option A for reduced storage in high-device-count scenarios.

### 8.8 Synchronization Flow

```
Client                                    Server
   │                                         │
   │──── Join room ─────────────────────────>│
   │                                         │
   │<─── Presence with ver=8 ────────────────│
   │                                         │
   │     [local ver=5, need sync]            │
   │                                         │
   │──── Get sender-keys ver=5 ─────────────>│
   │                                         │
   │<─── Delta (ver 6,7,8 changes) ──────────│
   │                                         │
   │     [verify signatures on keys]         │
   │     [generate sender key]               │
   │                                         │
   │──── Publish sender-key ────────────────>│
   │                                         │
   │<─── Result ver=9 ───────────────────────│
   │                                         │
```

## 10. Stanza Content Encryption (SCE) Integration

### 10.1 Overview

This specification RECOMMENDS integration with XEP-0420 (Stanza Content Encryption) for structured plaintext encryption. SCE provides a standard envelope format that preserves message semantics while encrypting sensitive content.

### 10.2 SCE Envelope Construction

When SCE is used, the plaintext to be encrypted is an SCE envelope:

```xml
<envelope xmlns='urn:xmpp:sce:1'>
  <content>
    <body xmlns='jabber:client'>Hello, room!</body>
    <markable xmlns='urn:xmpp:chat-markers:0'/>
  </content>
  <time stamp='2023-06-15T12:00:00Z'/>
  <from jid='alice@example.com/device1'/>
  <rpad>random-padding-here</rpad>
</envelope>
```

This envelope is then encrypted with the sender key as described in Section 8.4.

### 10.3 Affix Elements

The following affix elements MUST be included in SCE envelopes:
- `<time>`: Timestamp of message creation (replay mitigation)
- `<from>`: Sender's full JID (sender authentication)
- `<rpad>`: Random padding (traffic analysis mitigation)

### 10.4 Full Encrypted Message with SCE

```xml
<message to='room@conference.example.com' type='groupchat'>
  <encrypted xmlns='urn:xmpp:omemo:muc-sender-keys:0'>
    <header sid='123456' key-id='a1b2c3' counter='42'/>
    <payload>BASE64_OF_ENCRYPTED_SCE_ENVELOPE</payload>
  </encrypted>
  <store xmlns='urn:xmpp:hints'/>
</message>
```

### 10.5 Fallback for Non-SCE Implementations

Implementations not supporting SCE MUST encrypt the message body directly. In this case, the plaintext is simply the UTF-8 encoded message text. Receiving implementations SHOULD detect whether the decrypted content is an SCE envelope or raw text.

## 11. Message Archive Management (MAM) Considerations

### 11.1 Archiving Encrypted Messages

Servers implementing XEP-0313 (MAM) MUST store sender-key encrypted messages as-is. The server cannot decrypt the content, but must preserve:
- The `<encrypted>` element with header and payload
- All metadata (timestamp, sender, etc.)

### 11.2 MAM Query Responses

When a client queries MAM for a sender-key encrypted room:

```xml
<iq type='get' id='mam1'>
  <query xmlns='urn:xmpp:mam:2' queryid='q1'>
    <x xmlns='jabber:x:data' type='submit'>
      <field var='FORM_TYPE' type='hidden'>
        <value>urn:xmpp:mam:2</value>
      </field>
      <field var='with'>
        <value>room@conference.example.com</value>
      </field>
    </x>
  </query>
</iq>
```

The server returns archived messages with preserved encryption:

```xml
<message id='result1'>
  <result xmlns='urn:xmpp:mam:2' queryid='q1' id='archive-id-1'>
    <forwarded xmlns='urn:xmpp:forward:0'>
      <delay xmlns='urn:xmpp:delay' stamp='2023-06-15T12:00:00Z'/>
      <message from='room@conference.example.com/alice' type='groupchat'>
        <encrypted xmlns='urn:xmpp:omemo:muc-sender-keys:0'>
          <header sid='123456' key-id='a1b2c3' counter='42'/>
          <payload>BASE64_CIPHERTEXT</payload>
        </encrypted>
      </message>
    </forwarded>
  </result>
</message>
```

### 11.3 Decryption of Archived Messages

To decrypt archived messages, clients need the historical sender keys. Challenges:

1. **Key Availability**: If a sender has rotated their key, the old key is needed
2. **Counter State**: Client must track which counters have been used

**Recommendations:**

- Clients SHOULD cache previous sender keys for rooms they are members of
- Clients SHOULD store a mapping of key-id to key material
- When encountering an unknown key-id in MAM results, clients SHOULD:
  1. Check local key cache
  2. Request key from the sender via on-demand request
  3. Display "[Unable to decrypt historical message]" if key unavailable

### 11.4 Sender Key Retention

Publishers SHOULD retain the ability to re-distribute old sender keys for a reasonable period (e.g., 30 days) to support MAM catchup scenarios. This can be achieved by:

- Storing previous sender keys locally
- Responding to on-demand requests for old key-ids

## 12. Security Considerations

### 12.1 Forward Secrecy

This protocol provides two levels of forward secrecy:

1. **Per-message forward secrecy**: The chain key ratchet ensures that compromise of a message key does not reveal past message keys.

2. **Epoch-based forward secrecy**: Key rotation on membership changes provides forward secrecy boundaries. Compromise of a sender key before rotation does not reveal messages after rotation.

However, forward secrecy is weaker than Double Ratchet:
- Compromise of a chain key reveals all future messages until rotation
- Long periods between rotations increase exposure window

**Mitigations:**
- Mandatory rotation on membership changes
- Recommended periodic rotation (every 7 days or 1000 messages)
- Chain key ratchet for per-message forward secrecy

### 12.2 Key Compromise

If a member's sender key is compromised:
- Only that member's sent messages (since last rotation) are exposed
- Other members' messages remain protected
- Messages sent after the next rotation are protected

### 12.3 Server Trust

The server stores encrypted sender keys but cannot decrypt them. However:
- Server can observe room membership
- Server can observe message metadata (who sends when, message sizes)
- Server could withhold key updates (denial of service)
- Server could replay old key distributions (mitigated by signature verification)

### 12.4 Device Verification

Clients MUST verify OMEMO identity keys before trusting sender keys encrypted for their devices. The signature on the sender key payload MUST be verified against the sender's known OMEMO identity public key.

Clients SHOULD use existing OMEMO trust mechanisms:
- Trust On First Use (TOFU)
- Manual fingerprint verification
- Cross-signing / trust anchors

### 12.5 Replay Protection

Multiple layers of replay protection:

1. **Message counter**: Each message has a unique counter. Clients MUST reject:
   - Messages with counter less than the last processed counter (unless in skipped keys)
   - Messages with duplicate counters

2. **SCE timestamp**: When using SCE, the timestamp inside the encrypted envelope provides additional replay protection.

3. **AAD binding**: The counter is included in the AAD, so modifying the counter invalidates the authentication tag.

### 12.6 Malicious Member

A malicious room member could:
- Send garbage encrypted messages (recipients detect via AEAD failure)
- Refuse to rotate keys (detected by other members, can be kicked)
- Share their sender key outside the room (cannot be prevented cryptographically)

### 12.7 Denial of Service

Potential DoS vectors:
- Flooding with key rotation requests
- Publishing invalid sender keys
- Requesting sender keys repeatedly

**Mitigations:**
- Servers SHOULD rate-limit key operations
- Clients SHOULD cache sender keys and limit re-fetch attempts
- Invalid keys SHOULD be reported to moderators

### 12.8 Traffic Analysis

Even with encryption, metadata reveals information:
- Message timing
- Message sizes
- Sender identities

**Mitigations:**
- SCE random padding (`<rpad>`) obscures message sizes
- Clients MAY pad messages to fixed sizes
- Cover traffic is outside the scope of this specification

### 12.9 Rate Limiting

Servers SHOULD implement rate limiting on sender key operations to prevent abuse:

| Operation | Recommended Limit |
|-----------|-------------------|
| Key publication | 10 per minute per user per room |
| Key fetch (full) | 5 per minute per user per room |
| Key fetch (delta) | 30 per minute per user per room |
| On-demand key request | 20 per minute per user |

Servers MAY adjust these limits based on room size and server capacity. When rate limits are exceeded, servers SHOULD return:

```xml
<iq type='error' id='publish1'>
  <error type='wait'>
    <resource-constraint xmlns='urn:ietf:params:xml:ns:xmpp-stanzas'/>
    <text>Rate limit exceeded. Try again later.</text>
  </error>
</iq>
```

### 12.10 Key Revocation

If a user believes their sender key has been compromised before the normal rotation cycle:

1. The user SHOULD immediately generate and publish a new sender key
2. The user MAY notify room members via a groupchat message that previous messages should be considered potentially compromised
3. There is no formal revocation mechanism; the new key publication implicitly supersedes the old key

Implementations SHOULD provide a user-accessible "rotate key now" function for emergency situations.

### 12.11 Offline Device Handling

When a new member joins, existing members' sender keys are not yet encrypted for the new member's devices. This creates a window where the new member cannot decrypt messages.

**Recommendations:**

- Online members SHOULD re-publish their sender keys promptly when notified of new members
- New members SHOULD use on-demand key requests (Section 8.3) for faster key acquisition
- Servers MAY queue key update notifications for offline members
- Clients SHOULD display "[Waiting for encryption keys]" for messages that cannot yet be decrypted, and retry decryption when keys become available

### 12.12 Room Destruction

When a MUC room is destroyed:

1. Server MUST delete all stored sender keys for that room
2. Server MUST delete all version history for that room
3. Clients SHOULD delete locally cached sender keys for the destroyed room
4. No key rotation notification is sent (the room no longer exists)

### 12.13 MUC Private Messages

This specification applies only to groupchat messages within the MUC room. Private messages (MUC-PM) between room occupants are outside the scope of this specification.

For MUC-PM, implementations SHOULD:
- Use standard OMEMO (XEP-0384) encryption
- NOT use sender keys, as MUC-PM is a two-party conversation

### 12.14 Counter Persistence

Clients MUST persist the message counter to stable storage before sending each message. Failure to do so can result in counter reuse after a crash, which would:

1. Cause decryption failures for recipients (AAD mismatch)
2. Potentially leak message key material (same key + nonce reused)

**Recommendations:**

- Write counter to disk before calling the encryption function
- Use atomic write operations or write-ahead logging
- On recovery from crash, increment counter by a safety margin (e.g., +10) before resuming

## 13. Implementation Notes

### 13.1 Server Storage

Recommended database schema:

```sql
CREATE TABLE muc_sender_keys (
    room_jid VARCHAR(255) NOT NULL,
    user_jid VARCHAR(255) NOT NULL,
    key_id VARCHAR(64) NOT NULL,
    encrypted_keys JSONB NOT NULL,  -- {device_id: base64_blob}
    counter_hint INTEGER DEFAULT 0,
    updated_at INTEGER NOT NULL,
    deleted BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (room_jid, user_jid)
);

CREATE TABLE muc_sender_keys_version (
    room_jid VARCHAR(255) PRIMARY KEY,
    version INTEGER NOT NULL DEFAULT 0,
    last_rotation_at TIMESTAMP
);

CREATE TABLE muc_sender_keys_history (
    room_jid VARCHAR(255) NOT NULL,
    user_jid VARCHAR(255) NOT NULL,
    key_id VARCHAR(64) NOT NULL,
    encrypted_keys JSONB NOT NULL,
    valid_from INTEGER NOT NULL,
    valid_to INTEGER,
    PRIMARY KEY (room_jid, user_jid, key_id)
);

CREATE INDEX idx_sender_keys_updated 
    ON muc_sender_keys(room_jid, updated_at);
CREATE INDEX idx_sender_keys_version
    ON muc_sender_keys(room_jid, updated_at)
    WHERE deleted = FALSE;
```

The `muc_sender_keys_history` table is optional but recommended for supporting MAM decryption.

### 13.2 Client Storage

Clients MUST maintain per room:

```
Room State:
  - room_jid: string
  - key_version: integer (last synced server version)
  - own_sender_key: {
      key_id: string,
      chain_key: bytes[32],
      signature_key: bytes[32],
      counter: uint32,
      created_at: timestamp
    }
  - peer_keys: map of (jid → {
      key_id: string,
      chain_key: bytes[32],
      signature_public_key: bytes[32],
      counter: uint32,  -- last seen counter
      skipped_keys: map of (counter → message_key),
      updated_at: timestamp
    })
  - old_keys: list of {
      key_id: string,
      chain_key: bytes[32],
      expires_at: timestamp
    }
```

### 13.3 Graceful Degradation

If a client does not support sender keys:
- It will not see the `<sender-keys/>` element in presence
- It can fall back to standard per-message OMEMO
- Other clients can detect non-support via disco and include legacy OMEMO keys in messages

For mixed rooms (some clients support sender keys, some don't):

```xml
<message to='room@conference.example.com' type='groupchat'>
  <!-- Sender key encrypted for supporting clients -->
  <encrypted xmlns='urn:xmpp:omemo:muc-sender-keys:0'>
    <header sid='123456' key-id='a1b2c3' counter='42'/>
    <payload>BASE64_CIPHERTEXT</payload>
  </encrypted>
  <!-- Standard OMEMO for legacy clients -->
  <encrypted xmlns='urn:xmpp:omemo:2'>
    <header sid='123456'>
      <keys jid='legacy@example.com'>
        <key rid='99999'>BASE64_KEY</key>
      </keys>
    </header>
    <payload>BASE64_CIPHERTEXT</payload>
  </encrypted>
  <store xmlns='urn:xmpp:hints'/>
</message>
```

### 13.4 Performance Considerations

**Key Distribution Cost:**
- O(n) OMEMO encryptions when publishing a sender key
- Done once on join and on rotation (not per-message)

**Message Encryption Cost:**
- O(1) AES-256-GCM encryption
- One HMAC-SHA-256 for chain key ratchet

**Storage:**
- Server: O(n) encrypted blobs per room (n = members)
- Client: O(n) decrypted sender keys per room

**Bandwidth:**
- Key distribution: ~200 bytes per device
- Message overhead: ~50 bytes (header only, vs ~100 bytes per device for OMEMO)

For a 100-member room:
- OMEMO: ~10KB overhead per message
- Sender Keys: ~50 bytes overhead per message (99.5% reduction)

## 14. Business Rules

### 14.1 Room Configuration Changes

**Room becomes public (members-only disabled):**
1. Server MUST delete all sender keys
2. Server MUST broadcast key deletion notification
3. Clients MUST disable sender key encryption for the room
4. Messages fall back to standard OMEMO or plaintext

**Room becomes members-only:**
1. Server MAY enable sender key support
2. Existing members generate and publish sender keys
3. Normal protocol resumes

### 14.2 Member Affiliation Changes

**Member promoted to admin/owner:**
- No cryptographic changes required
- Key material remains the same

**Member demoted from admin/owner:**
- No cryptographic changes required
- Key material remains the same

**Member banned:**
- Same as member leaving: key rotation required

### 14.3 Error Recovery

**Client has stale keys:**
1. Decryption fails with authentication error
2. Client fetches fresh keys from server
3. Client retries decryption
4. If still failing, request on-demand from sender

**Client missed key rotation:**
1. Client receives `key-rotation-required` notification
2. Client rotates own key
3. Client fetches updated peer keys

**Server unavailable during rotation:**
1. Client queues rotation
2. Client retries with exponential backoff
3. Client continues using old key (with warning) until rotation succeeds

### 14.4 Counter Overflow

If a counter approaches 2^32:
1. Client MUST rotate sender key before overflow
2. Recommended rotation at 2^31 (2 billion messages)
3. New key starts with counter = 0

### 14.5 Concurrent Key Publication

If two devices of the same user try to publish sender keys simultaneously:
1. Server accepts the first publication
2. Second publication receives version conflict error
3. Second device fetches current state and either:
   - Uses the published key (if same user, Option B in 8.7)
   - Coordinates with first device (Option A in 8.7)

## 18. Examples

### 18.1 Complete Join Flow

Alice joins a room where Bob is already a member:

```xml
<!-- 1. Alice joins -->
<presence to='room@conference.example.com/Alice'>
  <x xmlns='http://jabber.org/protocol/muc'/>
</presence>

<!-- 2. Server sends presence with key version -->
<presence from='room@conference.example.com/Alice'>
  <x xmlns='http://jabber.org/protocol/muc#user'>
    <item affiliation='member' role='participant'/>
  </x>
  <sender-keys xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='3'/>
</presence>

<!-- 3. Alice fetches existing keys (she has none cached) -->
<iq type='get' to='room@conference.example.com' id='sk1'>
  <sender-keys xmlns='urn:xmpp:omemo:muc-sender-keys:0'/>
</iq>

<!-- 4. Server returns Bob's sender key, but NOT encrypted for Alice yet -->
<!--    (Bob published this before Alice joined, so no key for Alice's devices) -->
<iq type='result' from='room@conference.example.com' id='sk1'>
  <sender-keys xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='3'>
    <sender-key from='bob@example.com' key-id='bob-key-1'>
      <!-- No <key> element for Alice's device - Bob hasn't published for her yet -->
    </sender-key>
  </sender-keys>
</iq>

<!-- 5. Alice generates her sender key and publishes it -->
<!--    (She can't decrypt Bob's messages yet, but can start sending) -->
<iq type='set' to='room@conference.example.com' id='sk2'>
  <sender-key xmlns='urn:xmpp:omemo:muc-sender-keys:0' key-id='alice-key-1'>
    <!-- Alice's sender key, OMEMO-encrypted for her own device -->
    <key jid='alice@example.com' rid='alice-device-1'>OMEMO_ENCRYPTED</key>
    <!-- Alice's sender key, OMEMO-encrypted for Bob's device -->
    <key jid='bob@example.com' rid='bob-device-1'>OMEMO_ENCRYPTED</key>
  </sender-key>
</iq>

<!-- 6. Server confirms and returns new version -->
<iq type='result' from='room@conference.example.com' id='sk2'>
  <sender-key xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='4'/>
</iq>

<!-- 7. Server notifies Bob of update (Alice joined and published her key) -->
<message from='room@conference.example.com' to='bob@example.com'>
  <sender-keys-updated xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='4'/>
</message>

<!-- 8. Bob fetches delta to get Alice's sender key -->
<iq type='get' to='room@conference.example.com' id='sk3'>
  <sender-keys xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='3'/>
</iq>

<!-- 9. Server returns Alice's new key -->
<iq type='result' from='room@conference.example.com' id='sk3'>
  <sender-keys xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='4'>
    <sender-key from='alice@example.com' key-id='alice-key-1' updated='4'>
      <key rid='bob-device-1'>OMEMO_ENCRYPTED</key>
    </sender-key>
  </sender-keys>
</iq>

<!-- 10. Bob decrypts and verifies Alice's sender key -->
<!--     Bob now re-publishes his sender key to include Alice's devices -->
<iq type='set' to='room@conference.example.com' id='sk4'>
  <sender-key xmlns='urn:xmpp:omemo:muc-sender-keys:0' key-id='bob-key-1'>
    <key jid='bob@example.com' rid='bob-device-1'>OMEMO_ENCRYPTED</key>
    <!-- NEW: Bob's sender key, now also encrypted for Alice -->
    <key jid='alice@example.com' rid='alice-device-1'>OMEMO_ENCRYPTED_FOR_ALICE</key>
  </sender-key>
</iq>

<!-- 11. Server confirms -->
<iq type='result' from='room@conference.example.com' id='sk4'>
  <sender-key xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='5'/>
</iq>

<!-- 12. Alice receives update notification -->
<message from='room@conference.example.com' to='alice@example.com'>
  <sender-keys-updated xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='5'/>
</message>

<!-- 13. Alice fetches Bob's updated key and can now decrypt his messages -->
<iq type='get' to='room@conference.example.com' id='sk5'>
  <sender-keys xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='4'/>
</iq>

<iq type='result' from='room@conference.example.com' id='sk5'>
  <sender-keys xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='5'>
    <sender-key from='bob@example.com' key-id='bob-key-1' updated='5'>
      <key rid='alice-device-1'>OMEMO_ENCRYPTED_FOR_ALICE</key>
    </sender-key>
  </sender-keys>
</iq>

<!-- 14. Alice decrypts and verifies Bob's sender key -->
<!--     She verifies the Ed25519 signature against Bob's OMEMO identity -->
<!--     Alice can now decrypt Bob's messages -->
```

**Alternative: On-Demand Key Request**

Instead of waiting for Bob to re-publish (steps 10-14), Alice can request Bob's key directly:

```xml
<!-- Alice requests Bob's sender key directly -->
<iq to='bob@example.com' type='get' id='skreq1'>
  <sender-key-request xmlns='urn:xmpp:omemo:muc-sender-keys:0'
                      room='room@conference.example.com'>
    <device rid='alice-device-1'/>
  </sender-key-request>
</iq>

<!-- Bob responds with his sender key encrypted for Alice -->
<iq to='alice@example.com' type='result' id='skreq1'>
  <sender-key xmlns='urn:xmpp:omemo:muc-sender-keys:0'
              room='room@conference.example.com'
              key-id='bob-key-1'
              counter='42'>
    <key rid='alice-device-1'>OMEMO_ENCRYPTED_FOR_ALICE</key>
  </sender-key>
</iq>
```

This on-demand approach is faster when Bob is online, as Alice doesn't need to wait for Bob to notice the update notification.

### 18.3 Key Rotation After Member Leaves

```xml
<!-- Carol leaves the room -->
<presence from='room@conference.example.com/Carol'
          to='carol@example.com'
          type='unavailable'>
  <x xmlns='http://jabber.org/protocol/muc#user'>
    <item affiliation='none' role='none'/>
    <status code='110'/>
  </x>
</presence>

<!-- Server notifies remaining members -->
<message from='room@conference.example.com' type='groupchat'>
  <sender-keys-updated xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='10'/>
  <key-rotation-required xmlns='urn:xmpp:omemo:muc-sender-keys:0'
                         reason='member-left'
                         jid='carol@example.com'/>
</message>

<!-- Alice generates new sender key and publishes -->
<iq type='set' to='room@conference.example.com' id='rotate1'>
  <sender-key xmlns='urn:xmpp:omemo:muc-sender-keys:0' key-id='alice-key-2'>
    <key jid='alice@example.com' rid='alice-device-1'>OMEMO_ENCRYPTED</key>
    <key jid='bob@example.com' rid='bob-device-1'>OMEMO_ENCRYPTED</key>
    <!-- Note: no key for Carol -->
  </sender-key>
</iq>

<!-- Server confirms -->
<iq type='result' from='room@conference.example.com' id='rotate1'>
  <sender-key xmlns='urn:xmpp:omemo:muc-sender-keys:0' ver='11'/>
</iq>

<!-- Bob does the same rotation -->
```

### 18.5 Multi-Device Synchronization (Per-Device Keys)

```xml
<!-- Alice has two devices, each with their own sender key -->

<!-- Alice device 1 publishes -->
<iq type='set' to='room@conference.example.com' id='pub1'>

    <key jid='alice@example.com' rid='alice-device-1'>OMEMO_ENCRYPTED</key>
    <key jid='alice@example.com' rid='alice-device-2'>OMEMO_ENCRYPTED</key>
    <key jid='bob@example.com' rid='bob-device-1'>OMEMO_ENCRYPTED</key>

<!-- Alice device 2 publishes -->

    <key jid='alice@example.com' rid='alice-device-1'>OMEMO_ENCRYPTED</key>
    <key jid='alice@example.com' rid='alice-device-2'>OMEMO_ENCRYPTED</key>

</iq>

<!-- Messages from each device reference their own key -->
<message from='room@conference.example.com/Alice' type='groupchat'>

  </encrypted>
```

## 20. XML Schema

```xml
<?xml version='1.0' encoding='UTF-8'?>
<xs:schema
    xmlns:xs='http://www.w3.org/2001/XMLSchema'
    targetNamespace='urn:xmpp:omemo:muc-sender-keys:0'
    xmlns='urn:xmpp:omemo:muc-sender-keys:0'
    elementFormDefault='qualified'>

  <!-- Presence annotation -->
  <xs:element name='sender-keys'>
    <xs:complexType>
      <xs:sequence>
        <xs:element ref='sender-key' minOccurs='0' maxOccurs='unbounded'/>
      </xs:sequence>
      <xs:attribute name='ver' type='xs:nonNegativeInteger' use='optional'/>
      <xs:attribute name='full' type='xs:boolean' use='optional'/>
    </xs:complexType>
  </xs:element>

  <!-- Individual sender key -->
  <xs:element name='sender-key'>
    <xs:complexType>
      <xs:sequence>
        <xs:element ref='key' minOccurs='0' maxOccurs='unbounded'/>
      </xs:sequence>
      <xs:attribute name='from' type='xs:string' use='optional'/>
      <xs:attribute name='room' type='xs:string' use='optional'/>
      <xs:attribute name='key-id' type='xs:string' use='required'/>
      <xs:attribute name='counter' type='xs:nonNegativeInteger' use='optional'/>
      <xs:attribute name='updated' type='xs:nonNegativeInteger' use='optional'/>
      <xs:attribute name='deleted' type='xs:boolean' use='optional'/>
    </xs:complexType>
  </xs:element>

  <!-- Encrypted key for a device (OMEMO-encrypted blob) -->
  <!-- 
    The 'jid' attribute is:
    - OPTIONAL when inside <sender-key from='...'> (JID is inherited from parent)
    - REQUIRED when inside <keys> for inline distribution (no parent context)
    - REQUIRED when publishing a sender-key (identifies the recipient)
  -->
  <xs:element name='key'>
    <xs:complexType>
      <xs:simpleContent>
        <xs:extension base='xs:base64Binary'>
          <xs:attribute name='jid' type='xs:string' use='optional'/>
          <xs:attribute name='rid' type='xs:string' use='required'/>
        </xs:extension>
      </xs:simpleContent>
    </xs:complexType>
  </xs:element>

  <!-- On-demand key request (peer-to-peer) -->
  <xs:element name='sender-key-request'>
    <xs:complexType>
      <xs:sequence>
        <xs:element name='device' minOccurs='0' maxOccurs='unbounded'>
          <xs:complexType>
            <xs:attribute name='rid' type='xs:string' use='required'/>
          </xs:complexType>
        </xs:element>
      </xs:sequence>
      <xs:attribute name='room' type='xs:string' use='required'/>
    </xs:complexType>
  </xs:element>

  <!-- Update notification -->
  <xs:element name='sender-keys-updated'>
    <xs:complexType>
      <xs:attribute name='ver' type='xs:nonNegativeInteger' use='required'/>
    </xs:complexType>
  </xs:element>

  <!-- Rotation required notification -->
  <xs:element name='key-rotation-required'>
    <xs:complexType>
      <xs:attribute name='reason' type='xs:string' use='required'/>
      <xs:attribute name='jid' type='xs:string' use='optional'/>
    </xs:complexType>
  </xs:element>

  <!-- Counter hint update -->
  <xs:element name='sender-key-counter'>
    <xs:complexType>
      <xs:attribute name='key-id' type='xs:string' use='required'/>
      <xs:attribute name='counter' type='xs:nonNegativeInteger' use='required'/>
    </xs:complexType>
  </xs:element>

  <!-- Multi-device synchronization -->
  <xs:element name='sender-key-sync'>
    <xs:complexType>
      <xs:sequence>
        <xs:element name='device' minOccurs='0' maxOccurs='unbounded'>
          <xs:complexType>
            <xs:attribute name='rid' type='xs:string' use='required'/>
            <xs:attribute name='range-start' type='xs:nonNegativeInteger' use='required'/>
            <xs:attribute name='range-end' type='xs:nonNegativeInteger' use='required'/>
          </xs:complexType>
        </xs:element>
      </xs:sequence>
      <xs:attribute name='room' type='xs:string' use='required'/>
    </xs:complexType>
  </xs:element>

  <!-- Encrypted message -->
  <xs:element name='encrypted'>
    <xs:complexType>
      <xs:sequence>
        <xs:element ref='header'/>
        <xs:element ref='keys' minOccurs='0'/>
        <xs:element ref='payload'/>
      </xs:sequence>
    </xs:complexType>
  </xs:element>

  <xs:element name='header'>
    <xs:complexType>
      <xs:attribute name='sid' type='xs:string' use='required'/>
      <xs:attribute name='key-id' type='xs:string' use='required'/>
      <xs:attribute name='counter' type='xs:nonNegativeInteger' use='required'/>
    </xs:complexType>
  </xs:element>

  <!-- Optional inline key distribution for new members -->
  <xs:element name='keys'>
    <xs:complexType>
      <xs:sequence>
        <xs:element ref='key' maxOccurs='unbounded'/>
      </xs:sequence>
    </xs:complexType>
  </xs:element>

  <xs:element name='payload' type='xs:base64Binary'/>

</xs:schema>
```
